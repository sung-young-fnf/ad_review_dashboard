# Eval Guide — Binary Eval 작성법

> 스킬/Agent 품질을 측정하는 eval 기준을 작성하는 방법.
> 핵심: Yes/No만. 스케일(1~10) 금지.

---

## 왜 Binary인가?

스케일은 분산이 크다. 같은 출력을 두 번 채점하면 3점 → 5점으로 달라질 수 있다.
Binary는 일관성이 높다. "빌드 통과했는가?" → Yes 또는 No, 흔들림 없음.

eval 4개 × 실행 5회 = max 20점. 이 20점의 변화를 추적하면 개선 여부가 명확하다.

---

## Eval 유형별 예시

### Agent/스킬 출력 (코드 생성)

**나쁜 eval:**
- "코드가 깔끔한가?" → 주관적
- "Best practice를 따르는가?" → 어떤 best practice?
- "코드 품질 1~7점" → 스케일

**좋은 eval:**
- "pnpm tsc --noEmit가 에러 0개로 통과하는가?"
- "TODO/FIXME/placeholder 주석이 0개인가?"
- "모든 함수명이 동사로 시작하는가? (get, create, update, delete, ...)"
- "외부 호출(API, DB)마다 에러 핸들링이 있는가?"

### Agent/스킬 출력 (문서 생성)

**나쁜 eval:**
- "문서가 포괄적인가?" → 기준 불명
- "사용자 니즈를 충족하는가?" → 너무 열린 질문

**좋은 eval:**
- "필수 섹션(Goal, AS-IS, TO-BE, AC)이 모두 포함되었는가?"
- "모든 AC가 검증 가능한 구체적 조건인가? (모호한 '개선' 같은 표현 없음)"
- "Story 수가 3~8개 범위인가?"
- "이미 구현된 기능에 대해 '신규 구현' 표시가 없는가?"

### Agent/스킬 출력 (UI 변경)

**나쁜 eval:**
- "UI가 예쁜가?" → 주관적
- "사용성이 좋은가?" → 측정 불가

**좋은 eval:**
- "Route→Page→Component 추적으로 실제 렌더되는 컴포넌트를 수정했는가?"
- "shadcn/ui 컴포넌트를 사용했는가? (커스텀 HTML 금지)"
- "반응형 breakpoint(sm/md/lg)가 적용되었는가?"
- "aria-label 또는 접근성 속성이 인터랙티브 요소에 있는가?"

## 흔한 실수

### 1. eval이 너무 많음 (7개+)
스킬이 eval 기준을 앵무새처럼 반복하기 시작한다. 실제 품질 개선이 아니라 eval 게이밍.
**해결:** 3~6개로 줄이기. 가장 중요한 것만.

### 2. eval이 너무 좁음
"정확히 3개 bullet point" → 기술적으로 통과하지만 이상한 출력.
**해결:** 품질을 테스트, 구조를 강제하지 말 것.

### 3. eval이 겹침
"문법이 맞는가?" + "맞춤법이 맞는가?" → 이중 계산.
**해결:** 각 eval이 독립적인 영역을 테스트.

### 4. Agent가 측정할 수 없는 eval
"사람이 이걸 좋아할까?" → Agent는 항상 "Yes"라고 답함.
**해결:** 관찰 가능한 시그널로 변환. "좋아함" → "첫 문장에 구체적 수치/스토리가 있는가?"

---

## 3-Question Test

eval을 확정하기 전에:

1. **두 Agent가 같은 출력에 대해 동의할 수 있는가?**
   → No면 너무 주관적. 더 구체적으로.

2. **스킬이 이 eval만 게이밍할 수 있는가?**
   → Yes면 너무 좁음. 더 넓게.

3. **사용자가 이 기준을 실제로 신경 쓰는가?**
   → No면 삭제. 의미 없는 eval은 시그널을 희석.

## 템플릿

```
EVAL [N]: [짧은 이름]
Question: [Yes/No 질문]
Pass: [Yes의 구체적 조건 — 1문장]
Fail: [No가 되는 구체적 조건 — 1문장]
```

### 예시

```
EVAL 1: 빌드 통과
Question: 생성된 코드를 포함한 프로젝트가 pnpm tsc --noEmit를 에러 0개로 통과하는가?
Pass: 터미널 출력에 error가 0개
Fail: 1개 이상의 TypeScript 에러 발생

EVAL 2: BFF 완전성
Question: API 엔드포인트 추가 시 Backend + BFF Route + Frontend 호출이 모두 있는가?
Pass: 3개 레이어 모두 변경/추가됨
Fail: Backend만 있고 BFF 또는 Frontend가 누락

EVAL 3: 교정 없음
Question: 출력이 LEARNINGS.md의 기존 교정 패턴을 위반하지 않는가?
Pass: 알려진 교정 패턴 0개 위반
Fail: 이전에 교정된 패턴을 반복
```
