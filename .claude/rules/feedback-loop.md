## Mistake Feedback Loop (실수 지적 시 필수)
> WHY: "죄송합니다" → 바로 재작업 → 같은 유형 실수 반복 — 학습 없는 사과는 무의미
> 실수 → 원인 분석 → 교훈 저장 → 재작업 루프가 있어야 같은 실수가 0회로 수렴

**사용자가 실수/오류를 지적했을 때:**
1. **즉시 중단** — 변명/재시도 전에 멈추고 인정
2. **원인 분석 (5 Why)** — 아래 블록을 출력:
```
🔍 실수 분석
증상: [무엇이 잘못되었나]
직접 원인: [왜 발생했나]
근본 원인: [왜 그런 원인이 생겼나]
재발 방지: [구체적 체크리스트 또는 규칙]
교훈: [1문장 핵심]
```
3. **교훈 저장** — 반복 가능한 실수면 아래 중 하나에 기록:
   - `serena/write_memory` — 도메인별 교훈 영구 저장
   - `/compound` — 해결 패턴 문서화 (docs/solutions/)
   - `.claude/notes/{date}-{topic}.md` — 세션 교훈 노트
4. **LEARNINGS.md 플레이스홀더 채우기 (MANDATORY)** — Hook이 교정을 감지하면 LEARNINGS.md에 스켈레톤이 생성됨. 즉시 `Edit`으로 "(to be analyzed by agent)" 플레이스홀더를 실제 분석 결과로 교체:
   - `Wrong action`: 에이전트가 실제로 한 행동
   - `Correct action`: 사용자가 원한 행동
   - `Rule`: 재발 방지 규칙 (1문장)
5. **개선된 방법으로 재작업** — 원인 분석에서 도출된 체크리스트 적용 후 재시도

**트리거 키워드**: "왜 ~한 건가요?", "잘못 ~했잖아요", "아까 ~가 틀렸는데", "다시 해주세요"
**Hook 트리거**: `[Self-Improve] MANDATORY: 교정 감지됨` 메시지가 stderr에 나타나면 자동 실행

❌ 실수 지적에 "죄송합니다" 후 바로 재작업 = VIOLATION (원인 분석 필수)
❌ 원인 분석 없이 같은 접근법 재시도 = VIOLATION
❌ 교훈을 기록하지 않고 넘어감 = VIOLATION (반복 가능한 실수인 경우)
❌ LEARNINGS.md에 "(to be analyzed by agent)" 플레이스홀더가 남아있으면 = VIOLATION (즉시 채워야 함)

## Second-Try Retro (2번째 시도에 성공 시 필수)
> WHY: 1번째에 실패하고 2번째에 성공하면 "해결됐으니 끝"으로 넘어가기 쉬움.
> 하지만 1번째 실패 원인을 분석하지 않으면 다른 작업에서 같은 패턴이 반복됨.
> "2번만에 성공 = 프로세스에 구멍이 있다"는 신호.

**트리거**: 동일 요청에 대해 2번째 시도에서 성공 (1번째 수정이 불완전/미적용/미커밋)
**행동**:
1. **1번째 실패 원인 분석** — 아래 블록 출력:
```
🔄 Second-Try Retro
1차 시도: [무엇을 했나]
실패 원인: [왜 1차에서 완료되지 않았나]
누락된 단계: [빠뜨린 프로세스 스텝]
```
2. **프로세스 개선 판단** — 아래 중 해당하는 조치:
   - 기존 규칙에 체크리스트 항목 추가 → 해당 `.claude/rules/*.md` 직접 편집
   - 새 규칙이 필요하면 → `auto-promoted.md`에 EMERGING으로 추가
   - Hook으로 자동 감지 가능하면 → Hook 개선 제안
3. **교훈 기록** — `serena/write_memory` 또는 LEARNINGS.md에 기록

❌ 2번째 시도 성공 후 1차 실패 원인 분석 없이 넘어감 = VIOLATION
❌ 프로세스 개선 없이 "이번엔 커밋했으니 끝" = VIOLATION (구조적 방지책 필수)
